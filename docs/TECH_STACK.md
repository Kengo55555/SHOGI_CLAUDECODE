# 技術スタック説明書

本ドキュメントでは、将棋対戦Webアプリケーション「shogi-app」で採用した技術スタックと、それぞれの選定理由を説明する。

---

## 全体構成図

```
[クライアント (ブラウザ)]
    |
    |--- HTTP (REST API) ---> Next.js App Router (API Routes)
    |--- WebSocket ---------> Socket.IO Server
    |
[Next.js 16 (フルスタックフレームワーク)]
    |
    |--- ORM -------------> Prisma 7
    |
[PostgreSQL (Neon Serverless)]
```

---

## フロントエンド

### React 19 + Next.js 16

| 項目 | 内容 |
|------|------|
| パッケージ | `react@19.2.4`, `next@16.2.4` |
| 役割 | UI構築、ルーティング、サーバーサイドレンダリング |

**選定理由：**

- **App Router によるファイルベースルーティング**：対局画面 (`/game/[matchId]`)、棋譜閲覧 (`/records/[matchId]`)、認証 (`/login`, `/register`) など、多数の画面を持つ本アプリにおいて、ディレクトリ構造がそのままURLに対応するApp Routerは直感的かつ保守しやすい。
- **API Routes によるフルスタック開発**：`src/app/api/` 配下にREST APIを配置し、フロントエンドとバックエンドを同一リポジトリ・同一言語（TypeScript）で開発できる。認証API、対局API、通知APIなどすべてをNext.js内で完結させることで、デプロイやCI/CDの複雑さを排除した。
- **React 19 の Server Components**：対局一覧や棋譜閲覧など、初期表示の速度が重要な画面でサーバーサイドレンダリングを活用し、ユーザー体験を向上させている。
- **React 19 の最新機能**：React 19のフォームアクション等を活用し、認証フローなどのインタラクションをシンプルに実装できる。

### Tailwind CSS 4

| 項目 | 内容 |
|------|------|
| パッケージ | `tailwindcss@^4`, `@tailwindcss/postcss@^4` |
| 役割 | スタイリング |

**選定理由：**

- **高速なUI開発**：将棋盤・駒・持ち駒台・対局画面など、独自性の高いUIコンポーネントが多いため、ユーティリティファーストのアプローチで素早くスタイルを適用・調整できる。
- **レスポンシブデザイン**：PC・タブレット・スマートフォン対応が必須要件であり、Tailwindのレスポンシブユーティリティ（`sm:`, `md:`, `lg:`）で効率的に実現できる。
- **テーマカスタマイズ**：遊郭風ダークテーマなど、和風のカスタムテーマを柔軟に構築できる設計。CSSファイルの肥大化を防ぎつつ、伝統的な配色を表現している。
- **PostCSS統合**：`postcss.config.mjs` で Tailwind をPostCSSプラグインとして統合し、Next.jsのビルドパイプラインにシームレスに組み込んでいる。

### TypeScript 5

| 項目 | 内容 |
|------|------|
| パッケージ | `typescript@^5` |
| 役割 | 型安全な開発 |

**選定理由：**

- **将棋ロジックの型安全性**：駒の種類（`KomaType`）、盤面座標、手番（先手/後手）など、将棋特有のドメインモデルを型で厳密に表現することで、ルール実装のバグを未然に防止する。例えば、駒の動き定義や合法手生成の関数に不正な値が渡されることを型レベルで防げる。
- **フロントエンド・バックエンド共通コード**：`src/lib/shogi/core/` 配下の将棋ロジック（盤面操作、ルール判定、棋譜変換）をクライアントとサーバーの両方で共有しており、TypeScriptの型システムによって整合性を保証している。
- **API型定義の共有**：`src/types/` に定義された型（API レスポンス、Socket.IOイベント等）をフロントエンド・バックエンドで共有し、通信時の型不一致を防いでいる。

## バックエンド

### Next.js API Routes (Route Handlers)

| 項目 | 内容 |
|------|------|
| 配置 | `src/app/api/` 配下 |
| 役割 | REST API エンドポイント |

**選定理由：**

- **別途バックエンドサーバーが不要**：Express.js等の別フレームワークを立てる必要がなく、Next.jsだけで完結する。開発・デプロイのオーバーヘッドを最小化した。
- **ミドルウェアによる認証制御**：`src/lib/auth/middleware.ts` でセッション検証を一元管理し、各APIルートに適用している。
- **エンドポイント設計**：認証（`/api/auth/*`）、対局（`/api/matches/*`）、対戦募集（`/api/match-requests/*`）、通知（`/api/notifications/*`）と、RESTfulな構成で整理している。

### Socket.IO

| 項目 | 内容 |
|------|------|
| パッケージ | `socket.io@^4.8.3`, `socket.io-client@^4.8.3` |
| 役割 | リアルタイム双方向通信 |

**選定理由：**

- **将棋対戦のリアルタイム性**：対局中の指し手送受信は100ms以下のレイテンシが目標であり、HTTP pollingでは実現困難。WebSocketによる常時接続が必須である。
- **自動再接続・フォールバック**：Socket.IOはWebSocket接続が切断された場合の自動再接続機能を内蔵しており、対局中の接続断からの復帰をフレームワークレベルでサポートする。素のWebSocket APIでは自前実装が必要になる部分を省力化した。
- **イベント駆動アーキテクチャ**：`src/types/socket.ts` で定義された型付きイベント（指し手送信、対局状態同期、時間通知など）により、クライアント・サーバー間の通信プロトコルを明確に管理できる。
- **ルーム機能**：対局ごとのルーム（`matchId`ベース）で参加者を管理し、将来的な観戦機能（第三者がリアルタイム閲覧）の実装基盤にもなる。

## データベース

### PostgreSQL（Neon Serverless）

| 項目 | 内容 |
|------|------|
| パッケージ | `@neondatabase/serverless@^1.1.0`, `pg@^8.20.0` |
| 役割 | データ永続化 |

**選定理由：**

- **リレーショナルデータの整合性**：ユーザー、対局、棋譜、対戦募集、通知など、エンティティ間のリレーションが複雑であり、外部キー制約やトランザクションによるデータ整合性の担保が不可欠。NoSQLでは困難な一貫性保証をPostgreSQLが提供する。
- **JSONB型による棋譜保存**：`GameRecord.movesJson` フィールドにJSONB型で指し手配列を格納し、リレーショナルな構造と柔軟なJSON構造を一つのDBで両立している。将来的なクエリ（特定の局面検索等）にもJSONBのインデックスで対応できる。
- **Neon Serverless**：サーバーレス環境（Vercel等）との親和性が高い。コネクションプーリングをNeon側で管理するため、サーバーレス特有のコネクション枯渇問題を回避できる。`@prisma/adapter-neon` によりPrismaとシームレスに統合している。
- **UUID主キー**：全テーブルでUUID v4を主キーに採用し、分散環境でのID衝突を回避している。

### Prisma 7

| 項目 | 内容 |
|------|------|
| パッケージ | `prisma@^7.7.0`, `@prisma/client@^7.7.0` |
| 役割 | ORM / マイグレーション管理 |

**選定理由：**

- **スキーマ駆動開発**：`prisma/schema.prisma` にデータモデルを宣言的に定義し、マイグレーションとクライアントコードを自動生成する。将棋用語（`sente`, `gote`, `kifu` 等）を用いたモデル定義がそのまま型安全なクエリAPIになる。
- **型安全なデータアクセス**：`src/generated/prisma/` に生成される型付きクライアントにより、SQLの手書きミスやカラム名の誤りをコンパイル時に検出できる。
- **マイグレーション管理**：`prisma/migrations/` でスキーマ変更の履歴を管理し、開発・本番環境のDB同期を安全に行える。
- **Neonアダプター対応**：`@prisma/adapter-neon` により、Neon Serverless Driverを直接利用でき、エッジランタイムでも動作する。

## 将棋ゲームロジック

### 自前実装（`src/lib/shogi/`）

| 項目 | 内容 |
|------|------|
| 配置 | `src/lib/shogi/core/`, `src/lib/shogi/ai/` |
| 役割 | 将棋ルールエンジン、CPU対局AI |

**選定理由：**

- **完全なルール準拠**：二歩禁止、打ち歩詰め禁止、行き所のない駒の制御、千日手判定、持将棋判定など、将棋の全ルールを正確に実装する必要がある。既存のnpmパッケージでは将棋ルールの完全な実装が存在しないか、品質・保守性に課題があるため、自前実装とした。
- **クライアント・サーバー共有**：純粋関数として設計し、`src/lib/shogi/core/` をフロントエンド（合法手ハイライト、UX向上）とバックエンド（不正手検出、サーバー権威）の両方で使用する。
- **CPU AI**：`src/lib/shogi/ai/` に評価関数（`evaluate.ts`）と探索アルゴリズム（`search.ts`）を実装し、3段階のCPU対局レベルを提供。外部AIエンジンへの依存を排除している。
- **KIF形式棋譜出力**：`notation.ts` で日本語棋譜形式への変換を行い、将棋文化に準拠した棋譜保存・エクスポートを実現。

## 認証

### 独自トークン認証（パスワードレス）

| 項目 | 内容 |
|------|------|
| パッケージ | `crypto-js@^4.2.0`, `uuid@^13.0.0` |
| 配置 | `src/lib/auth/` |
| 役割 | ユーザー認証・セッション管理 |

**選定理由：**

- **パスワードレス認証**：`AuthToken` モデルによるメールベースのトークン認証を採用。パスワード漏洩リスクを根本的に排除し、ユーザーの負担も軽減する。
- **セッション管理**：`Session` モデルでサーバーサイドセッションを管理。トークンハッシュをDB保存し、セッションの無効化（ログアウト）を確実に行える。
- **外部認証サービス非依存**：Auth0やFirebase Auth等の外部サービスに依存せず、データの完全な制御権を保持。将来的なLINE連携やSlack連携（`NotificationSettings`で定義済み）への拡張も自前で対応可能。

| 項目 | 内容 |
|------|------|
| パッケージ | `eslint@^9`, `eslint-config-next@16.2.4` |

### Vitest

| 項目 | 内容 |
|------|------|
| パッケージ | `vitest@^4.1.4` |
| 役割 | ユニットテスト |

**選定理由：**

- **ESM対応**：プロジェクトのESモジュール構成とネイティブに互換性があり、設定の複雑さを最小限に抑えている。

## 技術選定の全体方針

| 方針 | 説明 |
|------|------|
| **フルスタックTypeScript** | フロント・バック・ゲームロジックすべてをTypeScriptで統一し、型定義の共有と開発効率の最大化を図った |
| **モノリポ構成** | フロントエンドとバックエンドを単一の`shogi-app`ディレクトリに集約し、デプロイと管理の簡素化を優先した |
| **サーバーレス対応** | Neon Serverless + Next.js の組み合わせにより、Vercel等へのゼロコンフィグデプロイが可能 |
| **将棋ドメイン最優先** | ルールエンジンと棋譜処理は外部ライブラリに頼らず自前実装し、将棋文化への正確な準拠を保証した |
| **リアルタイム性** | Socket.IOによる双方向通信で、対局体験に不可欠な即時性を確保した |
