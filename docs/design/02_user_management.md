## 1. 概要

パスワードレス認証（マジックリンク方式）による会員管理システム。メールアドレスのみで登録・ログインできるシンプルな構成とする。

---

## 2. 認証フロー

### 2.1 新規登録フロー

```
[ユーザー]                    [サーバー]                     [メール]
    │                            │                            │
    ├── メールアドレス入力 ──────→│                            │
    │                            ├── メール重複チェック        │
    │                            ├── ワンタイムトークン生成    │
    │                            ├── auth_tokens テーブルに保存│
    │                            ├── 確認メール送信 ──────────→│
    │                            │                            │
    │←── 「メールを確認してください」│                          │
    │                            │                            │
    │←── メール受信（リンククリック）────────────────────────────┤
    │                            │                            │
    ├── /api/auth/verify?token=xxx →│                         │
    │                            ├── トークン検証（有効期限・使用済み）
    │                            ├── usersテーブルに登録       │
    │                            ├── セッション発行            │
    │                            ├── notification_settings作成 │
    │←── ハンドルネーム設定画面 ──┤                            │
    │                            │                            │
    ├── ハンドルネーム送信 ──────→│                            │
    │                            ├── handle_name 更新          │
    │←── ダッシュボードへリダイレクト                           │
```

### 2.2 ログインフロー

```
[ユーザー]                    [サーバー]                     [メール]
    │                            │                            │
    ├── メールアドレス入力 ──────→│                            │
    │                            ├── ユーザー存在チェック       │
    │                            ├── ワンタイムトークン生成     │
    │                            ├── ログインリンクメール送信──→│
    │←── 「メールを確認してください」│                          │
    │                            │                            │
    │←── リンククリック ─────────────────────────────────────── ┤
    ├── /api/auth/verify?token=xxx →│                         │
    │                            ├── トークン検証              │
    │                            ├── セッション発行            │
    │←── ダッシュボードへリダイレクト                           │
```

### 2.3 セキュリティ考慮

- ワンタイムトークンの有効期限：**15分**
- トークンは1回使用で無効化
- トークンはSHA-256でハッシュ化して保存（DB漏洩対策）
- セッションの有効期限：**30日**（アクティビティで延長）
- セッションCookie: `HttpOnly`, `Secure`, `SameSite=Strict`
- 登録・ログインリクエストへのレート制限：5回/分/IP
- メールアドレス未登録でも「メールを送信しました」と返す（アカウント列挙防止）

## 3. データモデル

### 3.1 追加テーブル

#### auth_tokens（認証トークン）
```sql
CREATE TABLE auth_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email       TEXT NOT NULL,
  token_hash  TEXT NOT NULL,           -- SHA-256ハッシュ
  type        VARCHAR(10) NOT NULL,    -- 'register' | 'login'
  used        BOOLEAN NOT NULL DEFAULT false,
  expires_at  TIMESTAMPTZ NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_auth_tokens_hash ON auth_tokens(token_hash);
```

#### sessions（セッション）
```sql
CREATE TABLE sessions (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token_hash  TEXT NOT NULL UNIQUE,    -- SHA-256ハッシュ
  expires_at  TIMESTAMPTZ NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_used   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_sessions_token ON sessions(token_hash);
CREATE INDEX idx_sessions_user ON sessions(user_id);
```

## 4. API詳細設計

### 4.1 POST /api/auth/register

**リクエスト：**
```json
{
  "email": "user@example.com"
}
```

**処理：**
1. メールアドレスのバリデーション（形式チェック）
2. 既存ユーザーの有無を確認
   - 既存の場合：ログインリンクを送信（register と同じレスポンスを返す）
   - 新規の場合：registerタイプのトークンを生成
3. トークンをハッシュ化してauth_tokensに保存
4. メール送信（リンク: `{APP_URL}/auth/verify?token={token}&type=register`）

**レスポンス（常に200）：**
```json
{
  "message": "確認メールを送信しました"
}
```

### 4.2 GET /api/auth/verify

**クエリパラメータ：**
- `token`: ワンタイムトークン

**処理：**
1. トークンをSHA-256ハッシュ化して検索
2. 有効期限・使用済みチェック
3. トークンを使用済みに更新
4. type=register の場合：ユーザー作成 → notification_settings作成
5. セッション発行 → Cookie設定

**レスポンス：**
- 成功：ダッシュボードまたはハンドルネーム設定画面へリダイレクト
- 失敗：エラーページ表示（「リンクが無効または期限切れです」）

### 4.3 POST /api/auth/logout

**処理：**
1. セッショントークンをDBから削除
2. Cookie削除

### 4.4 GET /api/users/me

**レスポンス：**
```json
{
  "id": "uuid",
  "handleName": "将棋太郎",
  "email": "u***@example.com",
  "createdAt": "2026-04-19T00:00:00Z"
}
```

### 4.5 PATCH /api/users/me

**リクエスト：**
```json
{
  "handleName": "新しい名前"
}
```

**バリデーション：**
- ハンドルネーム：1〜20文字、空白のみ不可
- 禁止語フィルタ（不適切な名前の排除）

### 4.6 DELETE /api/users/me

**処理：**
1. 進行中の対局がないか確認（あればエラー）
2. ユーザーの論理削除（`deleted_at` 設定）
3. 全セッション削除
4. 対戦募集のキャンセル
5. 通知設定削除

**注意：** 棋譜データは対戦相手のためにも残す。ハンドルネームは「退会済みユーザー」に置換。

## 7. テスト観点

| テスト項目 | 内容 |
|-----------|------|
| 新規登録 | メール送信→リンククリック→ユーザー作成→セッション発行 |
| ログイン | 登録済みメール→リンク→セッション発行 |
| トークン期限切れ | 15分超過のトークンは無効 |
| トークン再利用 | 使用済みトークンは無効 |
| レート制限 | 6回目のリクエストが429で返ること |
| 退会 | 論理削除、セッション全削除、棋譜は残る |
| ハンドルネーム | バリデーション（長さ、空白） |
| アカウント列挙防止 | 未登録メールでも同じレスポンス |
