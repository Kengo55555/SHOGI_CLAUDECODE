## 1. プロジェクト概要

### 1.1 背景・目的
Web ブラウザ上で動作する将棋ゲームを開発する。ユーザーは一人で CPU（AI）と対戦することも、他の登録ユーザーとオンラインで対戦することもできる。対戦希望者が「対戦募集シグナル」を発信し、時間のある登録ユーザーが応諾することでマッチングが成立する、コミュニティ型の対戦プラットフォームを目指す。

### 1.2 サービスのコンセプト
- **気軽に一人で遊べる**：CPU対戦で棋力に応じた練習ができる
- **人と対戦できる**：登録しておくだけで、他の人から対戦の誘いを受けられる
- **自分の成長が見える**：戦績・棋譜・統計分析で棋力向上をサポート

### 1.3 運営形態
無料公開サービス（個人開発／非商用）

### 3.2 CPU対戦機能（F-04, F-05）

#### 3.2.1 AI レベル
| レベル | 強さの目安 | 実装方針 |
|-------|-----------|---------|
| 初級 | 将棋のルールを覚えたての人向け | 浅い探索（1〜2手読み）＋ランダム性 |
| 中級 | アマチュア級位者向け | 中程度の探索（3〜5手）＋簡易評価関数 |
| 上級 | アマチュア有段者向け | OSS将棋エンジン（例: やねうら王）をWeb Assembly等で組み込み |

#### 3.2.2 対局中の操作
- 投了ボタン
- 待った機能（CPU戦のみ・回数制限あり）
- 盤面リセット（途中終了）

---

### 3.3 オンライン対戦機能（F-06〜F-08）

#### 3.3.1 マッチング方式：対戦募集シグナル方式
他のオンライン対戦サービスとは異なり、本サービスでは **「募集ブロードキャスト → 応諾制」** を採用する。

**フロー：**
1. 対戦したいユーザー A が「対戦募集」ボタンを押す
2. 全登録ユーザーに通知（メール／LINE／Slack／サイト内通知）が送信される
3. 通知を見て時間のあるユーザー B が「応諾」ボタンを押す
4. A と B の対局が開始される
5. 先着1名が応諾した時点で募集は締め切られる

**ビジネスルール：**
- 同時に発信できる募集は1ユーザーあたり1件まで
- 募集の有効期限は30分（経過後自動キャンセル）
- 募集発信時に以下の条件を指定可能：
  - 持ち時間（10分 / 15分）
  - 手合（平手 / 駒落ちは2nd以降）

#### 3.3.2 対局進行
- WebSocket によるリアルタイム通信
- 双方の指し手を即時同期
- 接続断の場合は一定時間（例：60秒）待機し、復帰しなければ切断側の負けとする

### 3.4 通知機能（F-09〜F-11）

| 通知チャネル | 内容 | MVP |
|------------|------|:---:|
| サイト内通知 | ログイン時に未読通知を表示 | ◯ |
| メール通知 | 対戦募集あり／応諾された／対戦結果 | ◯ |
| LINE通知 | LINE Messaging API による通知 | |
| Slack通知 | Slack Incoming Webhook による通知 | |

**通知トリガー：**
- 他ユーザーが対戦募集を発信したとき（全員へ）
- 自分の募集が応諾されたとき（発信者のみ）
- 対局が終了したとき（両者へ）

**ユーザー設定：**
- 通知ON/OFF をチャネル毎に切り替え可能（重要・スパム化防止）

### 3.5 戦績管理機能（F-12〜F-15）

#### 3.5.1 記録する項目
- 対局日時
- 対戦相手（CPU の場合はレベル、人間の場合はハンドルネーム）
- 勝敗・決まり手（詰み／投了／時間切れ／接続断）
- 総手数
- 棋譜（KIF形式で保存）

#### 3.5.2 統計分析（2nd 以降）
- 勝率の推移グラフ
- 得意な戦型・苦手な戦型の判定
- よく指す手筋の集計
- 負けパターン分析（開始局面からの分岐）
- レーティング推移（イロレーティング方式）

#### 3.5.3 棋譜再生
- 1手ずつの早送り／巻き戻し
- 任意の局面からの検討（CPU思考を表示）

### 3.6 対局機能・持ち時間（F-16）

| モード | 内容 |
|-------|------|
| 10分切れ負け | 各プレイヤー持ち時間10分、使い切ったら負け |
| 15分切れ負け | 各プレイヤー持ち時間15分、使い切ったら負け |

**将棋ルール実装：**
- 本将棋（標準ルール）のみサポート
- 禁じ手の判定：二歩、打ち歩詰め、行き所のない駒
- 千日手・持将棋：MVPでは引き分け判定のみ（詳細なルールは2nd以降）

## 4. 非機能要件

### 4.1 性能要件
| 項目 | 目標値 |
|------|-------|
| ページ初期表示 | 3秒以内（一般的な回線） |
| 指し手反映 | 1秒以内（対戦相手の画面に反映） |
| CPU初級・中級思考時間 | 3秒以内 |
| CPU上級思考時間 | 10秒以内 |
| 同時接続数（MVP） | 50対局程度 |

### 4.2 セキュリティ要件
| 項目 | 対応 |
|------|-----|
| 認証 | メールアドレス＋ワンタイムリンク（パスワードレス方式を推奨） |
| 通信 | HTTPS 必須 |
| データ保護 | メールアドレスは暗号化して保管 |
| スパム対策 | 対戦募集の発信回数制限、reCAPTCHA 等の導入 |
| 不正対策 | 同一IPからの大量登録検知 |

> **注意**：「メールアドレスのみ」による認証は簡易だが、なりすましリスクがある。MVP では登録時にメール到達確認（ワンタイムリンク）を行うことを推奨。

### 4.3 可用性・運用要件
| 項目 | 内容 |
|------|-----|
| サービス稼働時間 | 24時間／365日（ベストエフォート） |
| バックアップ | DB日次バックアップ |
| 障害対応 | 個人運営のため、最大24時間以内の対応を目標 |

### 4.4 互換性・対応環境
| 区分 | 対応内容 |
|------|---------|
| PC ブラウザ | Chrome / Safari / Edge / Firefox の最新2世代 |
| スマホブラウザ | iOS Safari、Android Chrome の最新2世代 |
| レスポンシブ対応 | 必須（PC/タブレット/スマホでレイアウト自動調整） |

### 5.2 UI / ビジュアルデザイン要件

#### 5.2.1 駒のデザイン方針

**基本方針：伝統的な日本の将棋駒のスタイルを踏襲する**

| 項目 | 仕様 |
|------|------|
| 駒の形状 | 五角形（伝統的な将棋駒の形） |
| 駒の地色 | 木地を思わせる淡い黄褐色〜オレンジベージュ（例：#E8C88A 前後） |
| 通常駒の文字色 | 黒（墨色） |
| 成駒の文字色 | 赤（朱色） |
| 書体 | 毛筆体（楷書または行書）。成駒は草書寄りの崩し字も可 |
| 表現 | フラットなイラスト表現でも、木目テクスチャを入れた写実寄りでも可（統一感を優先） |

#### 5.2.2 駒デザインの参考イメージ
ユーザー提供の参考画像（伝統的な将棋駒イラスト）を基準とする。
- 王将・玉将・金将・銀将・桂馬・香車・歩兵・角行・飛車 は黒文字
- と金・成銀・成桂・成香・龍王・龍馬 は赤文字
- 背景は木地色、文字は毛筆体

#### 5.2.3 実装方針（アセット選定）
MVPでは以下のいずれかで実装する：

| 選択肢 | メリット | デメリット |
|-------|---------|-----------|
| ① OSS駒画像セット（CC0等）を利用 | 工数ゼロ・即利用可 | デザイン統一性に制約 |
| ② 伝統的駒の画像を調達（フリー素材／イラストAC等） | 品質高め・雰囲気出る | ライセンス確認必須 |
| ③ SVGで自作 | 自由度高・軽量・スマホでも綺麗 | デザインスキルと工数が必要 |

**推奨：MVPは ② または ③ を採用し、参考画像と同等の伝統的テイストを再現する。**
**注意**：ユーザー提供の参考画像（イラストAC等の有償・フリー素材サイトの画像）をそのまま使用する場合、必ず利用規約・ライセンスを確認すること。

#### 5.2.4 盤・駒台のデザイン
| 要素 | 仕様 |
|------|------|
| 盤の色 | 榧（かや）材を思わせる明るい木目色 |
| 盤の線 | 黒または濃茶の細線（9×9のマス目） |
| 駒台 | 盤と同系色、一段暗めの色味 |
| 最終手ハイライト | 半透明のカラーオーバーレイ |
| 合法手表示 | 移動可能マスに淡い色の●マーク |

#### 5.2.5 全体のデザイントーン
- **和の伝統感**と**現代Webの使いやすさ**を両立
- スマホでも駒の文字が潰れないよう、最小表示サイズを想定した可読性テストを行う
- カラーパレット：木地色・墨色・朱色を基調とし、UI要素（ボタン等）は落ち着いたアクセントカラーで補う

## 8. リリース計画

### MVP（1stリリース）：目標 1ヶ月以内
**必須実装：**
- メールアドレス登録（ワンタイムリンク認証）
- CPU対戦（3レベル・ただし上級は簡易実装版で妥協）
- 対戦募集シグナル方式のオンライン対戦
- メール通知＋サイト内通知
- 基本的な勝敗記録＋棋譜保存・再生
- 持ち時間（10分・15分）
- PC・スマホ両対応（レスポンシブ）

**この範囲に絞る理由：**
1ヶ月という期限内で「動くもの」を作るため、統計分析・レーティング・LINE/Slack連携・上級AIエンジン本格実装は2ndリリース以降とする。

### 2ndリリース：MVP後 1〜2ヶ月
- LINE / Slack 通知連携
- レーティング機能
- 統計分析（手筋・得意局面・勝率推移）
- CPU上級レベルの本格実装（OSSエンジン組み込み）

### 3rdリリース以降（将来構想）
- 駒落ち対戦
- 観戦機能・チャット機能
- 詰将棋問題機能
- トーナメント・リーグ戦機能

## 9. 制約・前提事項

- 個人開発・非商用のため、運用コストを最小化する構成を前提とする
- 個人情報はメールアドレスのみ収集し、最小限に留める
- 商用エンジン・有料APIは原則使用せず、OSS中心で構築する
- サーバー代はできる限り無料枠または低額プランでまかなう

## 10. リスクと懸念事項

| # | リスク | 影響度 | 対策 |
|---|-------|:------:|------|
| R-01 | 1ヶ月MVPスケジュールがタイト | 高 | 機能を絞り、統計分析等は2nd送り |
| R-02 | 対戦募集の通知がスパム化する可能性 | 中 | 通知ON/OFF設定、発信回数制限を設ける |
| R-03 | メール認証のみだと複数アカウント作成が容易 | 中 | IPベースのレート制限、不正検知 |
| R-04 | 同時応諾の競合（2人同時に応諾ボタンを押す） | 中 | DBトランザクションで先着1名に限定 |
| R-05 | 接続断による対局中断 | 中 | 再接続猶予期間＋自動勝敗判定ルール明記 |
| R-06 | CPU上級レベルの実装が難航 | 中 | MVPでは簡易版で代替、2ndで本格実装 |
| R-07 | ユーザー数が少ないと対戦が成立しない | 高 | MVPリリース時はCPU対戦を主機能として訴求 |
| R-08 | 登録ユーザー全員にメール通知するとメール送信コストが膨らむ | 中 | MVPはサイト内通知を主とし、メールはオプトイン推奨 |

## 11. 用語集

| 用語 | 説明 |
|------|------|
| 対戦募集シグナル | 対戦したいユーザーが全登録者に向けて発信する対戦募集通知 |
| 応諾 | 対戦募集を受けて対局を開始すること |
| 棋譜 | 対局の指し手の記録 |
| KIF形式 | 将棋の棋譜の標準テキストフォーマット |
| 切れ負け | 持ち時間を使い切った時点で負けとなるルール |
| イロレーティング | 対局結果から算出するプレイヤーの棋力指標 |
| MVP | Minimum Viable Product（最小実行可能プロダクト） |
